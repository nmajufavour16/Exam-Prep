Chapter One

Introduction to Operating Systems

1.0 — Two Definitions

Definition 1 — Program ViewAn OS is an elaborate set of programs developed by a programmer to control all operations within the computer, to such an extent that it does not require any human intervention. Three phases: Input → Process → Output.
Definition 2 — Hardware ViewAn OS is an Input Unit that translates tools to work with into the Computer (RAM = Random Access Memory, ROM = Read Only Memory). When you boot, ROM calls the OS, editors, compilers, assemblers, and interpreters.

1.1 — Four Components of an OS

Memory Manager

Tracks, allocates, and de-allocates all memory. Ensures programs get what they need.

Processor Manager

Manages CPU time. Schedules which process runs and for how long.

Device Manager

Manages all I/O hardware (printers, disks, keyboards) via device drivers.

File Manager

Handles file creation, deletion, reading, writing, and access permissions.

OS Functions — 4 Key Points

  • Keep track of the status of each resource (in use / free)
  • Monitor resources continuously
  • Enforce policies — who gets what resource
  • Allocate and de-allocate (reclaim) resources when appropriate

Chapter Two

Processes and Synchronization

2.1 — What is a Process?

DefinitionA process is a program in execution. It is a unit of work that requires resources — CPU time, memory, files, I/O devices — to accomplish its task. A program is passive (code on disk); a process is active (program + execution context).

Five Process States

StateDescription
NewProcess is being created
ReadyWaiting to be assigned to CPU
RunningInstructions are being executed
WaitingWaiting for an event (e.g. I/O completion)
TerminatedProcess has finished execution

Process Control Block (PCB)

The OS maintains a PCB for each process containing: process state, program counter, CPU registers, scheduling info, memory management info, accounting info, and I/O status info.

2.2.2 — Suspend and Resume

SuspendA process is moved out of main memory to secondary storage (disk) to free RAM. It cannot proceed until resumed. Resume = bring it back from disk to memory and continue execution.
  • Suspended processes are NOT in the ready queue
  • Caused by: memory overload, long wait, higher-priority process needs memory
  • Process retains its state — resumes exactly where it was suspended

2.2.3 — Interrupt Processing

DefinitionAn interrupt is a signal causing the CPU to suspend its current task, save its state, and handle a higher-priority event via an Interrupt Service Routine (ISR). "Another word for interrupt is After."
TypeCause / Example
Program InterruptSoftware error — e.g. dividing any number by ZERO
Timer InterruptSystem clock triggers → CMOS Battery
I/O InterruptI/O device completes or needs attention
Hardware FailureHardware component fails — power, memory error
EXAM FACT: Dividing any number by zero = Program Interrupt!

2.3 — The Nucleus (Kernel) of the OS

Nucleus / KernelThe core of the OS — runs with highest privilege, directly on hardware. Always resident in RAM (never swapped). Manages interrupt handling, process scheduling, and memory management.

2.5 — The Hierarchical Machine Concept

LayerComponentFunction
BottomHardwarePhysical CPU, memory, I/O devices
2KernelInterrupt handling, scheduling, I/O
3OS ServicesFile system, memory management
TopUser InterfaceCLI / GUI — user-facing programs

2.6 — Concurrent Processes

ConcurrencyTwo or more processes execute overlapping in time. On single-CPU: rapid context-switching (illusion of parallelism). On multi-CPU: true parallel execution. Concurrency requires synchronization.
  • Race Condition: Outcome depends on the order concurrent processes access shared data — unpredictable, a bug
  • Critical Section: Code that accesses shared resources; must NOT execute in more than one process at a time

2.7.1 — Semaphores

SemaphoreA synchronization tool (Dijkstra). An integer variable accessed only through two atomic operations: wait(S) — decrement S, block if S<0; and signal(S) — increment S, wake a waiting process.

Binary Semaphore

Value = 0 or 1. Used as a mutex lock — ensures only one process in critical section at a time.

Counting Semaphore

Value = 0 to N. Controls access to a pool of N identical resources.

2.7.2 — Monitors

MonitorA high-level synchronization construct — a module encapsulating shared data and the procedures operating on it. Only ONE process active inside at a time (automatic mutual exclusion). Uses condition variables with wait() and signal().
  • Safer and easier to use than semaphores
  • Developed by Hoare and Hansen
  • Used in Java's synchronized keyword

2.7.3 — Dining Philosophers Problem

Problem5 philosophers at a round table alternate between thinking and eating. One fork between each pair (5 forks). Eating requires BOTH left AND right forks. If all grab left fork simultaneously → circular wait → deadlock. No one eats.
  • Illustrates: deadlock (circular wait) and starvation
  • Solution 1: Allow only 4 philosophers at the table at once
  • Solution 2: Asymmetric — odd philosophers take left fork first, even take right fork first
  • Solution 3: Only allow picking up forks when BOTH are available simultaneously
Chapter Three

Deadlock & Storage Management

3.1 — Deadlock Occurrence

DefinitionA deadlock is a situation where a group of processes are permanently blocked — each holds some resources and waits for resources held by others — making it impossible for any process to proceed to completion.

Five Necessary Conditions (ALL must hold)

  • Mutual Exclusion — Only one process can use a resource at a time
  • Hold and Wait — A process holds resources while waiting to acquire more
  • No Preemption — Resources cannot be forcibly taken; only voluntarily released
  • Circular Wait — P0 waits for P1, P1 waits for P2, …Pn waits for P0
  • Irreversibility — System cannot self-resolve; deadlock is permanent without intervention

3.2.1 — Deadlock Prevention

Structurally eliminate at least one of the four necessary conditions before deadlock can occur.

Condition to EliminateHow
Mutual ExclusionMake resources shareable (only works for some resources like read-only files)
Hold and WaitRequire process to request ALL resources at once before starting
No PreemptionIf a process can't get resources, preempt its current resources
Circular WaitImpose total ordering on resources; processes must request in order

3.2.2 — Deadlock Avoidance: Banker's Algorithm

Banker's AlgorithmBefore granting a resource request, simulate the allocation and check if the system remains in a Safe State. Key formula: Need = Maximum Required − Currently Allocated. If Need ≤ Available → process can proceed.

Safe State

At least one process can run to completion, release resources, allowing others to proceed. System can satisfy all future requests.

Unsafe State

No process can complete with available resources. Does NOT necessarily mean deadlock — but could lead to it.

Worked Example:

ProcessAllocatedMaximumNeed (Max−Alloc)
P1572
P29112
P3484
Available = 2 units

P1 needs 2 ≤ 2 ✓ → P1 runs, finishes, returns 5+2=7 → Available = 9 → P2 needs 2 ✓ → P2 finishes → P3 needs 4 ✓. Safe sequence: P1→P2→P3. SAFE STATE

3.2.3 — Deadlock Detection

  • Allow deadlock to occur, then detect it
  • Use Resource Allocation Graph — if it has a cycle → possible deadlock
  • Use Wait-for Graph — edges show which process waits for which
  • Run detection algorithm periodically to find deadlocked processes

3.2.4 — Deadlock Recovery

  • Process Termination: Abort all deadlocked processes (drastic), OR abort one at a time until cycle is broken (costly)
  • Resource Preemption: Forcibly take resources from some processes and give to others. Need to handle rollback — return preempted process to safe state.

3.3.3/3.3.4 — Storage Placement Strategies

First Fit

Allocate the FIRST hole large enough. Fast. May leave small unusable fragments at start.

Best Fit

Allocate the SMALLEST hole large enough. Minimizes waste but must search entire list.

Worst Fit

Allocate the LARGEST hole. Leaves biggest remaining fragment for future requests.

Next Fit

Like First Fit but starts from last allocation point — more uniform distribution.

3.3.5 — Virtual Storage (Virtual Memory)

Virtual MemoryA technique allowing processes to execute even when NOT completely in main memory. Active pages are in RAM; the rest on disk. The OS swaps pages in/out as needed — giving the illusion of more memory than physically exists.
  • Logical address space can be much larger than physical memory
  • Page Fault: Needed page NOT in memory → OS loads it from disk → restarts instruction
  • Page Table: Maps logical page numbers to physical frame numbers

3.3.5.2 — Advantages of Virtual Memory

  • Programs can be larger than physical RAM
  • More processes run simultaneously
  • Increased CPU utilization and throughput
  • Less I/O needed — only load active pages

3.3.7 — Segmented Memory Management

SegmentationMemory divided into variable-length segments corresponding to logical units: code, data, stack, heap. Each has a segment number and offset. Segment table stores base address and limit for each segment.

3.5.2 — Psychological Trade-Offs

Fundamental OS design tension: Throughput (total work done per unit time) vs Response Time (how fast one user gets results). Optimizing for one usually harms the other. This guides all scheduling decisions.

3.6 — Thrashing

ThrashingA condition where a process/system spends MORE time swapping pages (paging) than executing instructions. CPU utilization drops sharply despite system being "busy." Caused by too many processes competing for too little RAM.
  • Solution 1: Reduce degree of multiprogramming — suspend some processes
  • Solution 2: Working Set Model — keep all pages a process actively uses (its working set) in memory
  • Solution 3: Increase physical RAM
Chapter Four

Memory Management — Early Systems

4.1 — Single User Configuration (SVC)

DefinitionA standalone computer running a batch OS (e.g. IBM OS/370, FORTRAN, MS-DOS). Only ONE job is in memory at a time. Job runs to completion before the next is loaded.

Advantages

  • Simple allocation strategy
  • Simple memory space
  • No complicated housekeeping

Disadvantages

  • Memory not fully utilized — wasted space
  • CPU idle during I/O operations
  • Only one user/job at a time

4.2 — Partitioned Memory Management

Fixed PartitioningMemory divided into fixed-size partitions at boot time. Each partition holds exactly one process. Simple but causes Internal Fragmentation — unused space inside a partition when process is smaller than partition.

4.3 — Dynamic Partitions

Dynamic PartitioningPartitions created dynamically — each exactly the size of the arriving process. More efficient but causes External Fragmentation — total free memory is sufficient but scattered as non-contiguous holes.
  • Compaction: Shuffle processes to consolidate free space — expensive, requires relocatable code

4.4 — Relocatable Dynamic Partitions

Relocatable MemoryPrograms can be moved (relocated) in memory during execution. Uses a Relocation Register: Physical Address = Logical Address + Base Register value. Handled by the MMU (Memory Management Unit).
Chapter Five

Memory Management — Recent Systems

5.1 — Paged Memory Allocation

PagingPhysical memory divided into fixed-size FRAMES; logical memory into same-size PAGES. Any page can go into any available frame — no need for contiguous physical memory. A page table maps page numbers to frame numbers.
  • Memory manager: (1) counts pages, (2) finds empty frames, (3) loads all pages into frames
  • Eliminates external fragmentation; may have internal fragmentation in last page
  • Pages measured in bytes (8 bits = 1 byte)

Example: Job = 650 lines, page size = 100 lines → 7 pages needed. Last page = 50 lines → 50 lines wasted = internal fragmentation.

5.2 — Demand Paged Memory Allocation

Demand PagingPages loaded into memory ONLY when needed (demanded) — not all at once. Uses a lazy swapper. When a needed page is not in memory → Page Fault → OS loads page from disk → restarts instruction.

Page Replacement Algorithms: FIFO (First-In-First-Out), LRU (Least Recently Used), Optimal (replace page not used for longest time — theoretical best).

5.3 — Segmented Memory Allocation

  • Memory divided into variable-length logical segments (code, data, stack, heap)
  • Each segment has a segment number and an offset within the segment
  • Segment table stores: base (starting physical address) + limit (length of segment)
  • Natural to programmer — matches logical program structure
  • Suffers from external fragmentation

5.4 — Segmented/Demand Paged Allocation

Combined SchemeEach segment is further divided into pages. Gets logical benefits of segmentation + physical benefits of paging. Eliminates external fragmentation while maintaining logical program structure. Used in Intel x86 protected mode.

Comparison of All Memory Schemes

RelocatableExternalCan compact; uses relocation register

Chapter Six

DOS — Disk Operating System

6.1 — Internal vs External DOS Commands

Internal Commands

Loaded into memory at boot. Always available without accessing disk.

DIR, CLS, COPY, DEL, REN, TYPE, DATE, TIME, VER, CD, MD, RD, PATH, SET, PROMPT

External Commands

Stored as .COM or .EXE files on disk. Loaded from disk each time used.

FORMAT, DISKCOPY, CHKDSK, FDISK, ATTRIB, XCOPY, SORT, FIND, BACKUP, RESTORE

6.2 — Files and File Names in DOS

  • File name MUST start with an alphabet
  • Name must reflect content of the file
  • Maximum 8 characters name + 3 characters extension (8.3 format)
  • Extension identifies type: .EXE (executable), .COM (command), .BAT (batch), .TXT (text)
  • Cannot use: / \ : * ? " < > |

Chapter Seven

Unix Operating System

7.1 — Brief History of Unix

  • Developed in 1969 at Bell Labs (AT&T) by Ken Thompson and Dennis Ritchie
  • Originally written in Assembly; rewritten in C (1973) — made it portable across hardware
  • First OS written in a high-level language
  • Foundation for: Linux, macOS, Android, iOS, BSD

7.2 — Unix Design Goals

  • Simple, elegant, and easy to use
  • Multi-user and multi-tasking from the ground up
  • Portable — written in C, runs on many hardware platforms
  • "Everything is a file" — devices, pipes, sockets all treated as files
  • Small focused programs that do one thing well, combined via pipes

7.3 — Unix Structure

LayerComponentRole
InnerKernelProcess mgmt, memory, I/O, file system — core OS
MiddleSystem CallsInterface between user programs and kernel
OuterShell / UtilitiesCommand interpreter (bash, sh, csh) + user programs

7.4 — Memory Management in Unix

  • Uses demand paging with page replacement
  • Supports swapping — entire processes swapped to disk when memory is low
  • Shared memory — multiple processes can map same physical pages
  • Copy-on-Write (COW): When process forks, pages only copied when one process modifies them

7.5 — Processor Management in Unix

  • Preemptive, priority-based scheduling
  • Time-sharing: each process gets a time quantum
  • Process priority decreases as it uses more CPU (prevents monopoly)
  • Supports foreground and background processes (& operator)

7.5.1 — Synchronization in Unix

  • Uses semaphores (System V IPC) for process synchronization
  • Uses signals for inter-process notification (SIGKILL, SIGTERM, SIGINT)
  • Supports message queues and shared memory segments

7.6 — Device Management in Unix

  • All devices are files — located in /dev directory
  • Block Devices: Transfer data in blocks — e.g. /dev/sda (hard disk)
  • Character Devices: Transfer character by character — e.g. /dev/tty (keyboard/terminal)
  • Device drivers are modules in the kernel

7.7 — File Management in Unix

Unix File SystemHierarchical tree starting at root (/). Everything is a file. Uses inodes to store file metadata: size, owner, permissions, timestamps, data block pointers. File names are case-sensitive.

7.7.1/7.7.2 — File Storage and File Names

  • File names are case-sensitive: File.txt ≠ file.txt
  • Can use almost any character except / (path separator) and null byte
  • No drive letters — everything under one root /
  • Hidden files start with a dot: .bashrc, .profile

7.7.3 — Pipes

Pipe ( | )Connects the output of one command to the input of another. Enables chaining specialized programs. Example: ls -l | grep ".txt" | sort — lists files, filters .txt files, sorts the result.

7.8 — Other Unix Features

  • Shell scripting: Automate tasks using bash/sh scripts
  • Permissions: Read(r), Write(w), Execute(x) for Owner, Group, Others — e.g. chmod 755
  • Symbolic Links: Shortcuts/aliases to files or directories
  • Redirection: > (output to file), < (input from file), >> (append to file)

7.9 — SCO Unix System

7.9.2 Multi-User

Multiple users log in and work simultaneously on one machine.

7.9.3 Multi-tasking

Multiple processes run concurrently via preemptive scheduling.

7.9.4 Shared Peripherals

Printers, modems, storage shared by all users.

7.9.5 Hierarchical File Structure

Tree-structured directories: / → /bin, /etc, /home, /var, /tmp.

7.9.8 Shell Environment

Command interpreter (bash, sh, ksh). Customizable via environment variables.

7.9.9 Security & Auditing

User authentication, file permissions (rwx), audit logs, stronger than DOS.

Linux Operating System

8.0 — Introduction

LinuxA free, open-source, Unix-like OS kernel created by Linus Torvalds in 1991 (Finland). Combined with GNU tools (Richard Stallman) it forms GNU/Linux — the OS used on servers, desktops, phones (Android), and supercomputers worldwide.

8.2 — Key Features of Linux

  • Fully open source — anyone can read, modify, distribute
  • Highly stable and secure — preferred for web servers
  • Hundreds of distros: Ubuntu, Fedora, Debian, CentOS, Kali, Arch
  • Advanced user and permission management (root = superuser)
  • Package managers: apt (Ubuntu/Debian), yum/dnf (RedHat), pacman (Arch)

Exam Prep — 25 Questions

Possible Exam Questions & Answers

TAP EACH QUESTION TO REVEAL THE ANSWER

Q01 — Ch.1 DefinitionDefine an Operating System. Give two definitions and state its three phases of operation.
Def 1 (Program View): An OS is an elaborate set of programs that controls all computer operations without requiring human intervention.

Def 2 (Hardware View): An OS is an Input Unit that translates tools into the computer (RAM/ROM). When you boot, ROM calls the OS, editors, compilers, assemblers, and interpreters.

Three Phases: Input → Process → Output.
Q02 — Ch.1 ComponentsName and explain the four components of an Operating System.
  • Memory Manager — Tracks, allocates, and de-allocates all memory
  • Processor Manager — Manages CPU time; schedules processes
  • Device Manager — Manages all I/O hardware via device drivers
  • File Manager — Handles file creation, deletion, reading, writing, and access permissions
Q03 — Ch.1 Multi-TermsDistinguish between Multi-programming, Multi-processing, and Multi-tasking.
Multi-programming: Multiple programs in memory simultaneously. CPU switches when one waits for I/O. Maximizes CPU use. No true parallelism on single CPU.

Multi-processing: Two or more CPUs in one system. True parallel execution. Increases throughput and reliability.

Multi-tasking: Rapid context-switching on single CPU giving illusion of simultaneous tasks. Time-shared CPU.
Q04 — Ch.2 ProcessesWhat is a process? Distinguish between a program and a process. List the five process states.
Process: A program in execution — active entity with allocated resources (CPU, memory, files).

Program vs Process: A program is passive (code on disk). A process is active (program + its execution state). Two processes can run the same program independently.

Five States: New → Ready → Running → Waiting → Terminated.
Q05 — Ch.2 Suspend/ResumeExplain Suspend and Resume in process management. When does the OS suspend a process?
Suspend: OS moves process from RAM to secondary storage (disk) to free memory. Suspended process is NOT in ready queue; cannot execute until resumed.

Resume: OS brings process back from disk into RAM; places it back in ready queue to continue from exactly where it was suspended.

Why suspend? Memory overload; process waiting too long; higher-priority process needs memory.
Q06 — Ch.2 InterruptsWhat is interrupt processing? Name and explain the four types of interrupts.
Definition: A signal causing the CPU to suspend its current task, save its state, and handle a higher-priority event via an ISR (Interrupt Service Routine). "Another word for interrupt is After."

  • Program Interrupt — Software error, e.g. dividing ANY number by zero
  • Timer Interrupt — System clock triggers → CMOS Battery
  • I/O Interrupt — I/O device completes or needs attention
  • Hardware Failure — Hardware component fails
KEY: Dividing by zero = Program Interrupt!
Q07 — Ch.2 NucleusWhat is the Nucleus (Kernel) of an OS? State its key responsibilities and one important characteristic.
Nucleus/Kernel: The innermost, most privileged core of the OS. Runs directly on hardware. Manages the most critical operations.

Responsibilities:
  • Interrupt handling
  • Process scheduling
  • Memory management
  • I/O management and device drivers
  • Manages user mode ↔ kernel mode transitions (system calls)
Key characteristic: ALWAYS resident in main memory — NEVER swapped out to disk.
Q08 — Ch.2 ConcurrencyDefine concurrent processes. What is a race condition and a critical section?
Concurrent Processes: Two or more processes executing overlapping in time. Single CPU: rapid context-switching. Multi-CPU: true parallel execution.

Race Condition: When the outcome depends on the order in which concurrent processes access shared data — result is unpredictable and non-deterministic. A serious bug.

Critical Section: Code that accesses shared resources and must NOT be executed by more than one process simultaneously. Solutions must satisfy: Mutual Exclusion, Progress, and Bounded Waiting.
Q09 — Ch.2 SemaphoresWhat is a semaphore? Distinguish between binary and counting semaphores. Name the two operations.
Semaphore: A synchronization integer variable (Dijkstra) accessed only via two atomic operations:

wait(S) / P(S): Decrement S. If S < 0, block the process.
signal(S) / V(S): Increment S. If S ≤ 0, wake a waiting process.

Binary Semaphore: Value = 0 or 1. Used as a mutex lock for critical sections.
Counting Semaphore: Value = 0 to N. Controls access to a pool of N identical resources.
Q10 — Ch.2 MonitorsWhat is a Monitor? How does it differ from a semaphore?
Monitor: A high-level synchronization construct — a module that encapsulates shared data and the procedures that operate on it. Only ONE process can be active inside at a time (mutual exclusion is automatic).

Difference from Semaphore:
  • Monitors are safer — mutual exclusion is built-in, not programmer responsibility
  • Monitors use condition variables (wait/signal on conditions)
  • Semaphores are lower-level and easier to misuse
  • Monitors used in Java (synchronized), Semaphores used in Unix (System V IPC)
Q11 — Ch.2 DiningDescribe the Dining Philosophers Problem. State two solutions and what OS concept it illustrates.
Problem: 5 philosophers at a round table. Each alternates thinking and eating. One fork between each pair (5 forks total). Eating needs BOTH forks. If all grab left fork simultaneously → circular wait → deadlock → no one eats.

Solutions:
  • Allow at most 4 philosophers at the table simultaneously
  • Asymmetric rule: odd philosophers pick left fork first; even pick right fork first
  • Use semaphore/monitor: only pick up forks when BOTH are available
Illustrates: Deadlock (circular wait), starvation, and resource allocation in concurrent systems.
Q12 — Ch.3 DeadlockDefine deadlock and state ALL five conditions necessary for it to occur.
Deadlock: A situation where a group of processes are permanently blocked — each holds some resources and waits for resources held by others — making it impossible for any process to proceed.

Five Conditions (ALL must hold simultaneously):
  • Mutual Exclusion — Only one process uses a resource at a time
  • Hold and Wait — A process holds resources while waiting for more
  • No Preemption — Resources cannot be forcibly taken; only voluntarily released
  • Circular Wait — Circular chain: P0 waits for P1, P1 for P2, …Pn for P0
  • Irreversibility — System cannot self-resolve the deadlock
Q13 — Ch.3 Deadlock MethodsWhat are the four methods of dealing with deadlock? Explain each.
  • Prevention — Structurally eliminate one of the four necessary conditions. E.g. require all resources requested upfront (eliminates hold-and-wait).
  • Avoidance — Use Banker's Algorithm. Only grant requests that leave system in a safe state.
  • Detection — Allow deadlock to occur. Use resource allocation graph/wait-for graph to detect it, then take action.
  • Recovery — After detection: terminate deadlocked processes one by one, OR preempt resources from processes and roll back to safe state.
Q14 — Ch.3 Banker'sExplain the Banker's Algorithm. What is a Safe State? Solve this: P1 alloc=3, max=9; P2 alloc=2, max=4; P3 alloc=3, max=7. Available=2.
Banker's Algorithm: Before granting a request, simulate allocation and verify the system remains in a safe state. Formula: Need = Max − Allocated. If Need ≤ Available → process can proceed.

Safe State: There exists a sequence where every process can complete and return resources to allow others to proceed.

Solution:
P1 Need = 9−3 = 6. P2 Need = 4−2 = 2. P3 Need = 7−3 = 4. Available = 2.
P2 needs 2 ≤ 2 ✓ → P2 runs, finishes, returns 2+2=4 → Available = 6
P3 needs 4 ≤ 6 ✓ → P3 finishes, returns 3+4=7 → Available = 13
P1 needs 6 ≤ 13 ✓ → SAFE. Safe sequence: P2 → P3 → P1.
Q15 — Ch.3 Virtual MemoryWhat is Virtual Memory? State its advantages and explain what a Page Fault is.
Virtual Memory: A technique allowing processes to execute even when NOT completely in main memory. Active pages are in RAM; the rest on disk. OS swaps pages in/out as needed — gives illusion of more memory than physically exists.

Advantages:
  • Programs can be larger than physical RAM
  • More processes run simultaneously
  • Increased CPU utilization and throughput
  • Less I/O — only load active pages
Page Fault: Occurs when needed page is NOT in RAM. OS suspends process, loads page from disk into a free frame, updates page table, then restarts the interrupted instruction.
Q16 — Ch.3 ThrashingWhat is Thrashing? What causes it and how is it resolved?
Thrashing: A condition where the system spends more time swapping pages than executing instructions. CPU utilization drops sharply despite system being "busy."

Cause: Too many processes in memory competing for too few frames. Each process causes constant page faults — loading one page evicts another that will soon be needed again.

Solutions:
  • Reduce degree of multiprogramming — suspend some processes to free memory
  • Working Set Model — ensure all pages a process actively uses are resident in memory
  • Increase physical RAM
Q17 — Ch.3 Storage StrategiesExplain the four storage placement strategies. Which is generally considered most efficient?
  • First Fit: Allocate first hole large enough. Fast but can fragment the start of memory.
  • Best Fit: Allocate smallest hole large enough. Minimizes wasted space per allocation but slow (searches all holes) and leaves many tiny fragments.
  • Worst Fit: Allocate largest hole. Leaves the biggest remaining fragment. Can be useful when future requests are large.
  • Next Fit: Like First Fit but starts from last allocation point. Distributes allocation more uniformly.
Generally: First Fit and Best Fit outperform Worst Fit. First Fit is often preferred for speed.
Q18 — Ch.4/5 Memory TypesCompare Early and Recent memory management techniques. List all types under each.
Early (Old) Memory Management:
  • Single User Configuration (SVC) — one job at a time; batch OS
  • Partitioned Memory Management — fixed-size partitions
  • Dynamic Memory Management — variable-size partitions
  • Relocatable Memory Management — programs moveable using relocation register
Recent (New) Memory Management:
  • Paged Memory Allocation — fixed-size pages/frames; page table
  • Demand Paged Memory Allocation — pages loaded only when needed; enables virtual memory
  • Segmented Memory Allocation — variable-length logical segments
  • Segmented/Demand Paged Allocation — combines segmentation and paging; best of both
Q19 — Ch.4 SVCWhat is Single User Configuration? State its characteristics, advantages, and disadvantages.
SVC: Standalone computer running a batch OS (IBM OS/370, FORTRAN, MS-DOS). One job in memory at a time; runs to completion before next job loads.

Characteristics: Keeps track of memory; obeys memory policy; allocates memory; de-allocates memory.

Advantages: Simple allocation strategy; simple memory space; no complicated housekeeping.

Disadvantages: Memory not fully utilized — wasted space; CPU idle during I/O; only one user/job at a time.
Q20 — Ch.5 FragmentationDistinguish between internal and external fragmentation. Give solutions for each.
Internal Fragmentation: Wasted space INSIDE an allocated block/page/partition — allocated memory is larger than what the process needs. Occurs with fixed partitions and paging (last page).
Example: 100KB partition given to 70KB process → 30KB wasted inside.
Solution: Smaller page/partition sizes (trade-off with overhead).

External Fragmentation: Total free memory is sufficient but scattered as non-contiguous holes — no single hole is large enough. Occurs with dynamic partitioning and segmentation.
Solutions: Compaction (expensive) or use Paging (which eliminates external fragmentation entirely).
Q21 — Ch.5 PagingExplain Paged Memory Allocation with a worked example. What is the page table?
Paging: Physical memory divided into fixed-size FRAMES; logical memory into same-size PAGES. Any page can go into any free frame — no contiguous requirement.

Before executing: (1) Count pages in program, (2) Find enough empty frames, (3) Load all pages into frames.

Example: Job = 650 lines, page size = 100 lines:
Page 1 → 100 lines, Page 2 → 100, Page 3 → 100, Page 4 → 100, Page 5 → 100, Page 6 → 50 lines (100−50 = 50 lines wasted = internal fragmentation). Total: 6 pages.

Page Table: Maps page number to frame number. OS uses it to translate every logical address to a physical address.
Q22 — Ch.6 DOS CommandsDistinguish between Internal and External DOS commands. Give five examples of each.
Internal Commands: Loaded into memory at boot — always available, no disk access needed.
Examples: DIR (list files), CLS (clear screen), COPY (copy files), DEL (delete file), REN (rename), CD (change directory), VER (version), DATE, TIME, TYPE

External Commands: Stored as .COM or .EXE files on disk — loaded from disk each time.
Examples: FORMAT (prepare disk), CHKDSK (check disk), XCOPY (extended copy), FDISK (disk partitioning), ATTRIB (file attributes), DISKCOPY, SORT, FIND
Q23 — Ch.6 Wild CardsWhat are wild cards in DOS? Explain * and ? with examples. Give rules for DOS file naming.
Wild Cards: Special characters representing unknown characters in file/directory names.

* (Asterisk) — Any number of characters:
DIR *.TXT → all .TXT files; DEL REPORT.* → all files named REPORT

? (Question Mark) — Exactly ONE character:
DIR FILE?.TXT → FILE1.TXT, FILE2.TXT but NOT FILE10.TXT

DOS File Naming Rules:
  • File name must start with an alphabet
  • Name must reflect the file's content
  • Maximum 8 characters name + 3 character extension (8.3 format)
  • Cannot use special characters: / \ : * ? " < > |
Q24 — Ch.7 UnixGive a brief history of Unix. State its design goals and describe its three-layer structure.
History: Developed 1969 at Bell Labs (AT&T) by Ken Thompson and Dennis Ritchie. Originally Assembly; rewritten in C (1973) — first OS in a high-level language, making it portable. Foundation for Linux, macOS, Android.

Design Goals: Simple and elegant; multi-user and multi-tasking; portable (written in C); "everything is a file"; small focused programs combined via pipes.

Three-Layer Structure:
1. Kernel (inner) — core OS: process management, memory, I/O, file system
2. System Calls (middle) — interface between user programs and kernel
3. Shell/Utilities (outer) — command interpreter (bash, sh, csh) + user programs
